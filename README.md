### Задание 1

Опишите основные преимущества использования масштабирования методами:

активный master-сервер и пассивный репликационный slave-сервер;
master-сервер и несколько slave-серверов;

####  Ответ  
##### активный master-сервер и пассивный репликационный slave-сервер  

Основные преимущества:  

1. Высокая доступность. Если Master выходит из строя, Slave может быть переведен в режим Master.  
2. Резервное копирование без остановки сервиса: Можно делать резервную копию БД на Slave-сервере. Это не создает дополнительной нагрузки на Master и не блокирует запись.
3. Разгрузка отчетовю Тяжелые аналитические запросы, отчеты или выгрузки данных можно направлять на Slave. Это защищает Master от высоких нагрузок на CPU/RAM из-за долгих чтений.  
4. Географическая репликация: Slave можно разместить в другом дата-центре. В случае катастрофы в основном ЦОДе, данные сохраняются в удаленной локации.


##### master-сервер и несколько slave-серверов

Это логическое развитие первого подхода, нацеленное уже не просто на отказоустойчивость, а на масштабирование производительности чтения.  

Основные преимущества:  

1. Линейное масштабирование чтения: Чем больше Slave-серверов будет добавлено тем больше параллельных запросов SELECT можно обслуживать.  
2. Балансировка нагрузки: Запросы на чтение распределяются между всеми Slave-серверами через балансировщик. Это предотвращает перегрев одной реплики и обеспечивает предсказуемое время ответа.  
3. Специализация серверов: Можно настроить разные Slave-серверы под разные задачи:  
   - Slave №1: Используется для пользовательских запросов.  
   - Slave №2: Используется для внутренней аналитики и отчетов.  
   - Slave №3: Используется исключительно для бэкапов.  
4. Меньшие риски при обновлении: Можно по очереди отключать Slave-сервера, обновлять на них версию СУБД или операционной системы, тестировать, а затем подключать обратно. Весь трафик чтения в это время пойдет на оставшиеся реплики. Master обновляется в последнюю очередь.

### Задание 2
Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц:

пользователи,
книги,
магазины (столбцы произвольно).
Опишите принципы построения системы и их разграничение или разбивку между базами данных.

Пришлите блоксхему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.

#### Ответ  

При подготовке ответа исходим из предпосылки что это дейтсвительно высоконагруженная БД а приложение которое общается с БД имеет микросервисную архитектуру.  

На рисунке представлена диаграмма решения:  

![Схема решения](https://github.com/cat2611/repl2-hw/blob/main/img/db-hw.drawio.png)

Для того чтобы сервис могвыдержать высокую нагрузку предлагается использовать следующий принцип при вертикльном шардировании:
Отказаться от разделения таблиц на шарды. Всместо этого вынести каждую таблицу в отдельный кластер СУБД. Таким образом мы получаем три кластера работа с которыми не затрагивает другие. Маршрутизация запросов идет на уровене микросервисов приложения.
Расмотрим каждый кластер:

##### Кластер Магазины(shops_cluster)  
Таблица магазины наименее загруженная. По этому использование шардинга будет лишним. На мой взгляд достаточно обычного решения с Master-сервером, который обслуживает запросы записи и изменения данных и реплики, обслуживающей запросы на чтение.  

##### Кластера Книги и Кластер Пользователи (books_cluster и users_cluster).  
На таблицы Книги и Пользователи будет основная нагрузка БД. По этому имеет смысл использовать на них горизонтальное масштабирование. Полем для определения шарда должно быть user_id и book_id в соответсвующих таблицах. Такой подход предотвратит перегрев шарда(Когда в одном шарде будет больше записей чем в другом). Количество Master-серверов должно определяться количеством Пользователей и книг. Из особенностей: Для обесепечения отказоустойчивости на каждый шард Master необходимо создавать сервер с несколькими репликами чтобы снять с Master-серверов операции чтения, а также иметь возможность выполнять балансировку этих запросов. Балансировка запросов записи и чтения должна определятся на уровне приложения.
